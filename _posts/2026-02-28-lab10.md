<p>
The behavior of malware that operate in the kernel space, such as rootkits, is not observable with user-level dynamic analysis tools like OllyDbg. To analyze malicious drivers and other kernel-level malware dynamically, a debugger must be used that can be attached to the OS itself. Chapter 10 of Practical Malware Analysis covers the use of WinDbg to accomplish this type of analysis.
</p>
<p>
The book uses a Windows XP VM for these labs, but I opted to use Windows 10 which requires a different setup that I covered in <a href="/malware/2026/02/27/network-debugging.html" target="_blank">this write-up</a>.
</p>
<p>
Also, the Window 7 drivers for Samples 1 and 3 from the <a href="https://github.com/mikesiko/PracticalMalwareAnalysis-Labs" target="_blank">PMA GitHub repo</a> had to be used. In the case of Sample 3, the Windows 7 driver still did not work and required modification.
</p>

<div id="1-2" class="section-link">
<h2>The user-space program calls the <b>ControlService</b> function. Can you set a breakpoint with WinDbg to see what is executed in the kernel as a result of the call to <b>ControlService</b>?</h2>
<p>
This sample includes an executable and a driver. Looking at the executable in IDA doesn't reveal much. It uses the driver to create a service and then start it. Next, <b>ControlService</b> is called with 1 as the <b>dwService</b> parameter, which stops the service:<br>
<img src="/images/lab10/executable.png"><br>
Nothing happens within the executable between starting the service and stopping it, which means that anything interesting is being performed with the driver at the kernel level.
</p>
<p>
Looking at the driver in IDA, there is also nothing suspicious going on in <b>DriverEntry</b>:<br>
<img src="/images/lab10/driver-entry.png">
</p>
<p>
Since there is nothing obvious happening within the main code of the driver, let's see what happens when <b>ControlService</b> issues the command to stop. This will require two instances of WinDbg; one in user-mode inside the VM to debug the executable and a second outside the VM to debug the kernel-mode execution of the driver. Trying to set a breakpoint on <b>ControlService</b> resulted in a "couldn't resolve" error because a stub function is being called instead of a call straight to ADVAPI32. The actual name can be discovered using the <b>x</b> command and a wildcard:<br>
<img src="/images/lab10/x.png">
</p>
<p>
Now, a breakpoint can be set on <b>ControlServiceStub</b> and execution can be resumed:<br>
<img src="/images/lab10/controlservicestub.png">
</p>
<p>
After hitting the user-mode breakpoint, the second instance of WinDbg can be started on the host machine outside of the VM. First, execution must be broken and then the driver object can be found using the <b>!drvobj</b> command:<br>
<img src="/images/lab10/drvobj.png">
</p>
<p>
Now that the memory location of the object is known, the details can be viewed with <b>dt _DRIVER_OBJECT</b>:<br>
<img src="/images/lab10/dt.png">
</p>
<p>
Since the goal is find out what happens when the stop command is issued, I set a breakpoint for the location of <b>DriverUnload</b> so that the code can be stepped through:<br>
<img src="/images/lab10/driverunload.png">
</p>
<p>
After resuming execution on both WinDbgs, several unknown calls are made:<br>
<img src="/images/lab10/calls-1.png">
</p>
<p>
To locate these calls in the static disassembly, I loaded the driver into IDA manually specifying a base using the <b>DriverStart</b> address found earlier with <b>dt _DRIVER_OBJECT</b> and jumped to the first call at <b>0xfffff801726e1022</b>:<br>
<img src="/images/lab10/calls-2.png">
</p>
<p>
Now we're getting somewhere! There are a series of calls creating registry keys and writing values. Checking the values that are written, it writes <b>0</b> to the values for the <b>DomainProfile</b>, <b>StandardProfile</b>, and base <b>WindowsFirewall</b> keys, disabling the firewall entirely:<br>
<img src="/images/lab10/values.png">
</p>
</div>

<div id="1-3" class="section-link">
<h2>What does this program do?</h2>
<p>
This program creates and starts a service using a malicious driver and then immediately stops it and terminates execution without performing any other actions. There is also no malicious code within the main <b>DriverEntry</b> section. However, upon unloading the driver it writes values to the registry which disable Windows Firewall. By performing these actions using a driver from within the kernel space, the need for administrator privileges is avoided and the process is invisible to user-mode applications.
</p>
</div>

<h2>Does this program create any files? If so, what are they?</h2>
<p>
The first call made in this sample is to <b>FindResourceA</b> with <b>0</b> as the <b>hModule</b> parameter, which searches the resources section of the current process, and <b>FILE</b> as the <b>lpType</b> parameter, indicating there is a file embedded within the resource section of this binary:<br>
<img src="/images/lab10/findresourcea.png">
</p>
<p>
When opened in Resource Hacker, evidence of a PE header can be seen with a quick glance at the hex editor:<br>
<img src="/images/lab10/reshacker.png">
</p>
<p>
After exporting the resource as a binary it can then be viewed in IDA. The entry point is labeled as <b>DriverEntry</b>, which means this is a driver:<br>
<img src="/images/lab10/driver.png">
</p>
<p>
Back to the executable; after finding the resource, it is then loaded with <b>LoadResource</b> and written to <b>C:\Windows\System32</b> as a file named <b>Mlwx486.sys</b>:<br>
<img src="/images/lab10/mlwx486.png">
</p>

<div id="2-2" class="section-link">
<h2>Does this program have a kernel component?</h2>
<p>
Taking a closer look at the driver embedded in the resource section of the executable, it begins by initializing a unicode string. I needed to look at IDA's hex view to see the complete value:<br>
<img src="/images/lab10/ntquerydirectoryfile.png">
</p>
<p>
Next, <b>MmGetSystemRoutineAddress</b> is called twice; once with <b>NtQueryDirectoryFile</b> and again with <b>KeServiceDescriptorTable</b>:<br>
<img src="/images/lab10/mmgetsystemroutineaddress.png">
</p>
<p>
<b>KeServiceDescriptorTable</b> is the kernel structure where pointers to system calls are stored. <b>NtQueryDirectoryFile</b> is a Windows function that returns file information. After locating these in memory, there is a loop that iterates over the <b>KeServiceDescriptorTable</b> structure until it locates <b>NtQueryDirectoryFile</b>. After locating the function, the loop is broken and a function from within this program is moved into that location, replacing the system call:<br>
<img src="/images/lab10/replace.png">
</p>
<p>
The replacement function begins by performing <b>NtQueryDirectoryFile</b> as normal. If only one file is returned, execution jumps to the end of the function and <b>NtQueryDirectoryFile</b>'s normal return value is used. However if multiple files are returned such as when obtaining a list of files in a directory, a comparison is performed between the first 8 bytes of each file's information at offset 0x5E (file name) and a string stored in the .text section:<br>
<img src="/images/lab10/cmp.png">
</p>
<p>
This is a UTF-16 string, which is 2 bytes per character, meaning the first four characters of the string are compared to the file name:<br>
<img src="/images/lab10/utf-16.png">
</p>
<p>
Looking at the raw hex, those characters are <b>Mlwx</b>. This routine checks to see if the file name being returned by <b>NtQueryDirectoryFile</b> matches the <b>Mlwx486.sys</b> driver that was written to <b>C:\Windows\System32</b>:<br>
<img src="/images/lab10/mlwx.png">
</p>
<p>
If there is a match, <b>bl</b> is incremented and the instruction that adds the file's information to the <b>FileInformation</b> structure that <b>NtQueryDirectoryFile</b> returns is skipped, excluding it from the results:<br>
<img src="/images/lab10/match.png">
</p>
</div>

<div id="2-3" class="section-link">
<h2>What does this program do?</h2>
<p>
This program extracts a binary from its resources section which is then used to write a driver to <b>C:\Windows\System32</b>. That driver is then used in the creation of a new service which hooks system calls to <b>NtQueryDirectoryFile</b> and replaces them with code that will hide the malicious driver from directory listings. I like to run these samples and analyze the results to verify my findings, but unfortunately the service creation failed for me on both Windows XP and Windows 10.
</p>

<p>

<p>
After creating the device, <b>DeviceIoControl</b> is called to send it a control code of <b>0xABCDEF01</b> and the output of that operation is directed to the local variable IDA has named <b>BytesReturned</b>:<br>
<img src="/images/lab10/deviceiocontrol.png">

<p>
Now, <b>OleInitialize</b> is called followed by <b>CoCreateInstance</b>, which creates a handle to a Windows COM object. The <b>rclsid</b> parameter passed to the call belongs to Internet Explorer:<br>
<img src="/images/lab10/rclsid.png">

<img src="/images/lab10/addtype.png">

<div id="3-3" class="section-link">
<h2>What does the kernel component do?</h2>
<p>
Fortunately, WinDbg was a little more cooperative with this sample and I was able to set a breakpoint on <b>DeviceIoControl</b>:<br>
<img src="/images/lab10/deviceiocontrolbp.png">
</p>
<p>
After hitting the user-mode breakpoint within the VM, the driver can be examined in the kernel-mode instance on the host machine. However, the <b>!drvobj</b> with the name of the driver file returns nothing:<br>
<img src="/images/lab10/noobject.png">
</p>
<p>
But remember that a device named <b>ProcHelper</b> was created with <b>CreateFileA</b>, the handle to which was used in the <b>DeviceIoControl</b> call. Here, the <b>!devobj</b> command can be used instead:<br>
<img src="/images/lab10/devobj.png">
</p>
<p>
Now <b>dt nt!_DRIVER_OBJECT</b> can be used with the driver object address that was returned for that device:<br>
<img src="/images/lab10/prochelperobject.png">
</p>
<p>
Using the <b>dq</b> command to display data from memory in quad-word form (since this is the 64-bit driver) along with the object base address plus the offset to <b>MajorFunction</b> returns a list of function addresses (L1C returns 0x1C results):<br>
<img src="/images/lab10/dq.png">
</p>
<p>
Only three functions were shown in <b>MajorFunction</b>. Most were at <b>0xfffff8017cb1b9c0</b>, which is just error handling:<br>
<img src="/images/lab10/iopinvaliddevicerequest.png">
</p>
<p>
The <b>ln</b> command was not useful for the other functions because they are defined within the object and have no associated labels. The best option at this point is to manually load the driver in IDA with the same base address as the <b>DriverStart</b> address <b>0xfffff801b1a70000</b> obtained from the <b>dt nt!_DRIVER_OBJECT</b> call earlier. The only call within the first function at <b>0xfffff801b1a71000</b> is to <b>IofCompleteRequest</b>, which runs any routine specified by the <b>Irp</b> parameter upon request completion:<br>
<img src="/images/lab10/iofcompleterequest.png"><br>
Setting a breakpoint on that function does not yield any results before crashing, so this function is never reached. This may be due to the fact that this sample crashes before completing requests on 64-bit systems, but this function still does not appear to do anything malicious. However the opportunity exists to pass a malicious function pointer to <b>IofCompleteRequest</b> and this knowledge may be useful in the future.
</p>
<p>
The final function at <b>0xfffff801b1a71020</b> <i>does</i> have some code worth looking at. This function gets the address of the current process and rewrites some values at two offsets from that address. These offsets are meant to point into the <b>ActiveProcessLinks</b> structure of the current process. Because each process is sized differently there is no way to track processes using fixed offsets. The solution to this in Windows is storing pointers to the next process and previous process in the <b>ActiveProcessLinks</b> structure of each process, which forms a circular double-linked list between all the running processes. The offset to this structure varies widely depending on Windows version. This driver was written for Windows 7 and results in a crash on my Windows 10 VM. I patched the driver so that it will work on Windows 10 and documented the process in <a href="/malware/2026/03/21/driver-patch.html" target="_blank">this write-up</a>.
</p>
<p>
In Windows 7, the <b>ActiveProcessLinks</b> structure is located at the offset <b>0x188</b> from the process address. The structure is a list of just two items: <b>flink</b> (forward link) and <b>blink</b> (back link). The first part of this function stores <b>flink</b> in RCX. Adding 8 to that address results in a pointer to <b>blink</b>, which is stored in R8:<br>
<img src="/images/lab10/activeprocesslinks.png">
</p>
<p>
Next, the <b>flink</b> for the current process is moved into the <b>flink</b> for the previous process, overwriting the address of the current process. Then, the address of <b>flink</b> is moved into RCX and <b>blink</b> is moved into RAX. The address of <b>blink</b> is written into <b>flink</b> + 8, which is the <b>blink</b> of that process. What this does is overwrite references to the current process in the <b>ActiveProcessLinks</b> structures of the adjacent processes, removing the current process from the linked list. This is the routine that hides the malicious process from view:<br>
<img src="/images/lab10/overwrite.png">
</p>
<p>
It's difficult to understand this process without visualizing it. Each process has its own <b>flink</b> and <b>blink</b>. The process ahead of the current process holds a pointer to the current process in its <b>blink</b>. The process behind the current process holds a pointer to the current process in its <b>flink</b>. The red arrows in this graphic try to illustrate where the overwrites are occurring:<br>
<img src="/images/lab10/graphic.png">
</p>
