<div id="1-1" class="section-link">
<h2>How can you get this malware to install itself?</h2>
<p>
This malware is installed by passing the <b>-in</b> argument when running the program, which installs it as a service. If there are 3 arguments in the command (filename, -in, password), the service is created using the filename. If there is a fourth argument (filename, -in, servicename, password), that is used as the service name. The book questions for this lab are a bit out of order and how this option is arrived at will be covered in the next question.<br>
<img src="/images/lab9/service.png">
</p>
</div>

<div id="1-2" class="section-link">
<h2>What are the command-line options for this program? What is the password requirement?</h2>
<p>
First, I looked at how to avoid the self-deletion command, which was quickly identified by looking at some of the parameters in the subroutine <b>0x402410</b> and verifying by stepping through with OllyDbg:<br>
<img src="/images/lab9/self_delete.png">
</p>
<p>
It looks like to avoid this function, the program needs to see at least one argument passed to it:<br>
<img src="/images/lab9/jumps.png">
<ol>
    <li>Jumps to the code where arguments are analyzed</li>
    <li>Jumps to the self-deleting function</li>
    <li>Jumps to a JMP that ends execution</li>
</ol>
</p>
<p>
After verifying that at least one argument has been passed, the last argument is pushed to the stack and <b>0x402510</b> is called. If this call doesn't return <b>1</b> to EAX, the self-deletion function is called again:<br>
<img src="/images/lab9/check_pw.png"><br>
Two things can be inferred from this series of instructions: The password is passed as the final argument and the subroutine at <b>0x402510</b> verifies that it is correct.
</p>
<p>
I dove a bit deeper than required to answer this question and attempted to reverse-engineer the password-checking function. One of the cool features of OllyDbg that I haven't seen in similar programs is that you can open files with arguments:<br>
<img src="/images/lab9/args.png">
</p>
<p>
Using "testpass" as a test password, I set a breakpoint to the beginning of the function so that I could retry other passwords easily and started stepping through it. The first check that takes places is a comparison of ECX and a hard-coded value of 4:<br>
<img src="/images/lab9/ecx-4.png"><br>
</p>
<p>
In this case, it was comparing against the value 8, so the check failed:<br>
<img src="/images/lab9/ecx-8.png"><br>
After testing some other passwords, it turns out that this is a length check, meaning the password is 4 characters.
</p>
<p>
Next, each character of the password is checked. The first and third are easily identified as <b>a</b> and <b>c</b>, but the second and fourth use a series of hard-to-follow calculations:<br>
<img src="/images/lab9/ac.png">
</p>
<p>
After looking at this, I tried a password of <b>abcd</b> and was successful! I still wanted to see how it could be figured out without a guess though. Trying different characters and looking at the results of the character-checking operations in OllyDbg quickly leads to figuring out the needed difference:<br>
<img src="/images/lab9/s.png"><br>
Trying a password of <b>asdf</b>, the operation for the second character results in 12. To get the <b>1</b> needed in ECX to pass the check, subtracting 0x11 from 0x73 (s) results in 0x62, or <b>b</b>.
</p>
<p>
After making it past the password check, there are four commands that can be passed as arguments:<br>
<ul>
    <li><b>-in</b> - Installs the malware by copying the executable to <b>C:\Windows\System32</b>, creating a service using that executable, and creating a new registry key</li>
    <li><b>-re</b> - Uninstalls the malware</li>
    <li><b>-c</b> - Modifies the registry values that were created (more on this in Question 5)</li>
    <li><b>-cc</b> - Prints the current registry value</li>
</ul>
</p>
<p>
There is also a fifth option, which is no argument. For this option to work, the malware must have already been installed. I will go into detail about how this works in Question 5.
</p>
</div>

<div id="1-4" class="section-link">
<h2>What are the host-based indicators of this malware?</h2>
<p>
Following the <b>-in</b> command (with the optional name argument), the malware is first copied into <b>C:\Windows\System32</b> and then a service is created using the optional name with " Manager Service" appended and the copied file:<br>
<img src="/images/lab9/manager-service.png">
</p>
<p>
Later in this function some parameters, including a malicious URL, are pushed to the stack and the <b>0x401070</b> subroutine is called:<br>
<img src="/images/lab9/url.png">
</p>
<p>
This function creates the key <b>HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft \XPS</b>. Its value is difficult to follow statically and will be easier to check during dynamic analysis, but it is a safe assumption that the values pushed to the stack before this function was called are what is written to the key:<br>
<img src="/images/lab9/key.png"><br>
From this analysis, the best host-based indicator would be the look-alike Microsoft key with a space appended and its XPS sub-key. If the original filename is known, <b>C:\Windows\System32</b> can be checked for its presence as well as any services pointing to it. The service's display name can vary depending on the argument chosen or the filename, but will always include " Manager Service".
</p>
</div>

<div id="1-5" class="section-link">
<h2>What are the different actions this malware can be instructed to take via the network?</h2>
<p>
Going back to the three jump cases at the beginning of the main function, there is one case I skipped in order to answer the other questions:<br>
<img src="/images/lab9/jumps-2.png"><br>
If no arguments are passed and the first jump is not taken, the subroutine at <b>0x401000</b> is called, which checks for the presence of the malicious registry entry discussed earlier. If that key is present, the call will be successful and the second jump will also not be taken. Before the third jump, a call is made to the function at <b>0x402360</b>. Thinking about this for a minute is important because the key is present, meaning this malware has been installed already, and now it is executing a new function with no arguments.
</p>
<p>
The function at <b>0x402360</b> has two important subroutines at <b>0x401280</b> and <b>0x402020</b>. The first reads values from the malicious registry key and these values are then used as arguments for the second. The <b>0x402020</b> subroutine then contacts the C2 server, parses the response, compares the values to five commands: SLEEP, UPLOAD, DOWNLOAD, CMD, and NOTHING. If <b>0x402020</b> returns 0, the loop begins again and otherwise exits. The four default values are <b>ups</b>, <b>http://www.practicalmalwareanalysis.com</b>, <b>80</b>, and <b>60</b> (notice the second Microsoft key with the trailing space):<br>
<img src="/images/lab9/regc-1.png"><br>
The <b>ups</b> parameter does not seem to have a function. The next two are the C2 URL and port, and the last parameter is the frequency that the C2 is contacted. After installation, a new C2 URL can be specified with the <b>-c</b> flag which will be contacted at the specified interval to retrieve one of the SLEEP, UPLOAD, DOWNLOAD, CMD, or NOTHING commands.
</p>
<p>
Just to see this in action, I set up a dummy C2 on an Ubuntu VM and directed the victim machine's outgoing traffic to it using ApateDNS. After installing the malware using the <b>-in abcd</b> arguments and then running the sample again with no arguments, the dummy C2 receives the following request:<br>
<img src="/images/lab9/c2.png">
</p>
</div>

<h3>Conclusion</h3>
This chapter gave me a lot of good practice with dynamic analysis using OllyDbg. Using dynamic analysis, I was able to avoid the tedious process of manually reverse engineering a deobfuscation routine and letting the program decode the C2 URL for me. This was a great way to learn my way around OllyDbg so that I can view dynamic data such as where DLLs have actually be loaded in memory and also allows me to verify findings from static analysis. I also learned how to patch binaries so that in certain cases like password protection, the password does not need to be known at all!
